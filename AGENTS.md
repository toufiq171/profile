# AGENTS.md

Repository guidance for coding agents working in this project.

## 1) Project Overview

- Project type: static personal portfolio website.
- Hosting target: GitHub Pages.
- Main stack: HTML + CSS + static JS assets.
- No package manager or app framework is used.
- Primary editable files:
  - `index.html`
  - `style.css`
  - `README.md`
  - `gitserver`
- Most files under `assets/` are vendor-generated and should be treated as external/static assets.

## 2) Repository Layout

- `index.html`: single-page content and structure.
- `style.css`: project-level overrides and simplification styles.
- `assets/`: third-party CSS/JS/fonts/images from Mobirise export.
- `.github/workflows/static.yml`: deploy workflow for GitHub Pages.
- `gitserver`: helper CLI for health/build/publish automation.
- `site-agent.sh`: legacy helper script (keep unless asked to remove).
- `.build/`: local build output (generated by `gitserver build`).

## 3) Source of Truth

- Prefer edits in `index.html` and `style.css`.
- Avoid editing minified vendor assets unless user explicitly asks.
- Keep text/content updates in `index.html`, and visual system updates in `style.css`.
- Preserve existing section IDs used by navbar anchors.

## 4) Build / Lint / Test Commands

This repo has no Node/Python test framework by default.
Use the commands below as the standard agent workflow.

### Build

- Build static output:
  - `./gitserver build`
- Output directory:
  - `.build/`

### Lint / Quality Checks

- Built-in health checks:
  - `./gitserver health`
- Placeholder link check:
  - `grep -n "mobiri.se\|mobirise.eu\|info@site.com" index.html`
- Quick HTML sanity check (no external deps):
  - `python3 - <<'PY'`
  - `from pathlib import Path; s=Path('index.html').read_text()`
  - `assert '<html' in s and '</html>' in s and '<title>' in s`
  - `print('html sanity ok')`
  - `PY`

### Test (Manual + Scripted Smoke)

- Local preview:
  - `python3 -m http.server 8080`
- Open and verify:
  - page loads, nav links scroll, contact link opens mail client.

### Single Test (Important)

Use this when asked to run only one test:

- Single check for contact email wiring:
  - `python3 - <<'PY'`
  - `from pathlib import Path; s=Path('index.html').read_text()`
  - `assert 'mailto:toufiqhamid171@gmail.com' in s`
  - `print('single test passed: contact email')`
  - `PY`

## 5) Deployment / Publish

- Normal publish path:
  - `./gitserver run "<commit message>"`
- This does:
  - pull/rebase from `origin/main`
  - stage all changes
  - commit
  - push to `origin/main`
- GitHub Actions deploys Pages from `main` via `.github/workflows/static.yml`.

## 6) Code Style Guidelines

### HTML

- Keep semantic tags (`section`, `nav`, `h1`-`h5`, `p`, `ul/li`).
- Keep IDs stable (e.g., `#header05-1`, `#form02-6`) to avoid breaking anchors.
- Use meaningful alt text; avoid placeholder alt values.
- Use absolute HTTPS links for external profiles.
- Keep copy concise and personal; remove template/editor instructions.

### CSS

- Put custom styles in `style.css`, not vendor CSS.
- Prefer small override blocks instead of mass rewrites.
- Use kebab-case for class names and CSS variables.
- Keep responsive typography with `clamp()` where appropriate.
- Avoid large `!important` usage unless overriding framework display classes.

### JavaScript

- No custom JS framework is present.
- If adding JS, keep it minimal and progressive-enhancement friendly.
- Avoid introducing bundlers unless user asks for a stack migration.
- Prefer plain browser APIs and defer scripts when possible.

### Imports / External Dependencies

- Do not add package managers by default.
- Reuse existing static assets under `assets/` first.
- Add new CDN dependencies only with a clear reason.
- If adding a dependency, document why in `README.md`.

### Naming Conventions

- HTML/CSS classes and IDs: kebab-case.
- JS variables/functions: camelCase.
- Files: lowercase with dashes when creating new files.
- Keep existing filenames unchanged unless asked.

### Formatting

- Preserve existing formatting style per file.
- Do not run broad auto-format across vendor-generated HTML/CSS.
- Keep lines readable; avoid huge one-line HTML inserts.

### Types / Data

- For JS additions, validate expected types from DOM inputs.
- Coerce form values carefully (`trim`, null checks).
- Do not assume optional elements exist; guard queries.

### Error Handling

- Fail safely: if optional UI elements are missing, no hard crash.
- Wrap network calls in `try/catch` and show user-facing fallback text.
- Keep error messages actionable and concise.

## 7) Agent Operating Rules for This Repo

- Default branch is `main`.
- Assume this is production-facing (public profile site).
- Prefer small, reviewable commits.
- Do not delete media/assets unless explicitly requested.
- Do not rewrite full page structure when only copy/layout tweaks are requested.
- Always run at least `./gitserver health` before finishing.

## 8) Autopilot Mode (Minimal Direction)

When user asks for a full autonomous update, agents should:

1. Run `./gitserver health`.
2. Read `index.html` + `style.css` and remove obvious template leftovers.
3. Improve clarity, spacing, and mobile readability without changing overall identity.
4. Keep external links and email values consistent.
5. Run `./gitserver build`.
6. Commit and push with `./gitserver run "Refine website content and layout"`.
7. Return what changed and the expected Pages URL.

If the user says "autopilot" or "do it end-to-end", execute the above sequence.
